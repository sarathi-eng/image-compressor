## 2024-05-29 - Missing File Size Limits and Error Leakage in Image Compression API
**Vulnerability:** The image compression API endpoint lacked file size validation before buffering the file into memory using `file.arrayBuffer()`. This exposed the application to Denial of Service (DoS) attacks via memory exhaustion. Additionally, the error handling block returned the exact `error.message` from the compression process to the client, potentially leaking internal implementation details (e.g. from the 'sharp' library).
**Learning:** Next.js API routes handling binary data (like file uploads) must explicitly enforce size limits before allocating memory for the buffer, as `arrayBuffer()` reads the entire file into memory at once. Furthermore, exposing raw error messages to the client violates the "Fail Securely" principle and can provide attackers with insights into the system's architecture.
**Prevention:** Always implement strict file size validation (e.g., using `file.size`) before calling `arrayBuffer()` or similar memory-intensive operations. For error handling, catch errors, log the details internally for debugging, and return a generic, non-informative error message (e.g., "Compression failed") to the client.
