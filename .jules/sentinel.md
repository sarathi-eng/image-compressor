## 2024-05-22 - [DOS via Image Processing Memory Exhaustion & Information Leakage]
**Vulnerability:** The `/api/compress` endpoint accepted image files of any size and any type into memory (`Buffer.from(await file.arrayBuffer())`), which could be exploited by an attacker to cause an Out-Of-Memory (OOM) crash, leading to a Denial of Service. In addition, the catch block returned `error.message` from the `sharp` library which could potentially leak stack traces or internal paths to the user.
**Learning:** `sharp` processing in Next.js API routes loads entire files into memory buffers. When file boundaries are missing, this creates an easy OOM attack vector. Furthermore, libraries like `sharp` often include internal path information in error messages, which must be scrubbed before returning a 4xx response to the client.
**Prevention:** Always implement explicit maximum file size limits (e.g. 15MB) and validate `file.type.startsWith("image/")` before allowing a file to be processed or loaded into memory. Also, implement generic error messages for the user response, while logging the complete error internally.
